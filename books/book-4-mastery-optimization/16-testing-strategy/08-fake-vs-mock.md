# Fake vs Mock

## Looking Back

In the ViewModel test, we injected a `FakeUserRepository`.

```kotlin
private val fakeUserRepository = FakeUserRepository()

viewModel = HomeViewModel(fakeUserRepository)
```

A natural question appears:

```text
What Is A Fake?

When Do We Use A Mock Instead?
```

---

## The Core Distinction

```text
Fake

↓

A Real Implementation, Simplified For Tests

Has Actual Logic

Uses In-Memory Data Instead Of Database Or Network

-------------------------

Mock

↓

A Generated Object That Records Calls

Has No Logic

You Tell It Exactly What To Return For Each Call
```

---

## The Same Problem, Two Solutions

Suppose we need to test a ViewModel that depends on `UserRepository`.

We cannot use the real `UserRepository`:

```text
Real UserRepository

↓

Calls Retrofit

↓

Hits Real Network

↓

Tests Are Slow, Flaky, Require Internet
```

We need a substitute.

---

## A Fake

```kotlin
class FakeUserRepository : UserRepository {

    private val users =
        MutableStateFlow<List<User>>(emptyList())

    fun emit(users: List<User>) {
        this.users.value = users
    }

    override fun observeUsers(): Flow<List<User>> = users

    override suspend fun refresh() {
        // no-op in tests
    }
}
```

The Fake:

```text
Implements the same interface

↓

Has real logic (stores users in memory)

↓

We control what it returns

↓

No network, no database
```

---

## A Mock

```kotlin
val mockRepository = mock<UserRepository>()

whenever(mockRepository.observeUsers())
    .thenReturn(flowOf(listOf(user)))
```

The Mock:

```text
Generated by a library (Mockito, MockK)

↓

No logic of its own

↓

You specify exactly what each call returns

↓

Also records which methods were called
```

---

## When To Use Each

### Use a Fake when:

```text
The dependency has complex behavior you want to reuse

You need to emit multiple values over time (Flow)

Multiple tests share the same fake

The test logic reads more naturally with a real-ish object
```

---

### Use a Mock when:

```text
You need a one-off return value for a single test

The dependency has many methods and you only care about one

You want to verify a specific method was called

Setting up a full Fake would take more time than the test is worth
```

class FakeUserRepository : UserRepository {

## Production Code — A Reusable Fake

```kotlin
class FakeUserRepository : UserRepository {

    private val _users =
        MutableSharedFlow<List<User>>(replay = 1)

    private var shouldThrow = false

    fun emit(users: List<User>) {
        _users.tryEmit(users)
    }

    fun setError(shouldThrow: Boolean) {
        this.shouldThrow = shouldThrow
    }

    override fun observeUsers(): Flow<List<User>> = _users

    override suspend fun refreshUsers(): Result<Unit> {
        return if (shouldThrow) {
            Result.failure(IOException("Network error"))
        } else {
            Result.success(Unit)
        }
    }
}
```

### How To Read It

```text
MutableSharedFlow(replay = 1)

↓

Stores the last emitted value

New collectors immediately receive the current state

-------------------------

emit(users)

↓

Test controls what flows through the repository

Call this before collecting to set up the scenario

-------------------------

setError(shouldThrow = true)

↓

Switches the fake into error mode

Tests the error path without touching the network

-------------------------

Result.failure(IOException(...))

↓

Matches the contract of the real implementation

Fake and real share the same interface
```

## The Fake Lives In testFixtures

```text
src/
├── main/
│   └── kotlin/
│       └── UserRepository.kt        ← real implementation
└── testFixtures/
    └── kotlin/
        └── FakeUserRepository.kt    ← shared test double
```

`testFixtures` is a Gradle source set.

Both unit tests and integration tests can share the same Fake.

---

## Fakes Catch Interface Drift

Suppose we add a method to `UserRepository`:

```kotlin
interface UserRepository {
    fun observeUsers(): Flow<List<User>>
    suspend fun refreshUsers(): Result<Unit>
    suspend fun deleteUser(id: String)  // ← new method
}
```

The Fake breaks immediately:

```text
FakeUserRepository is not abstract

and does not implement abstract member: deleteUser()
```

The Mock does not break.

It silently returns `null`.

```text
Fake

↓

Breaks when the interface changes

You fix it immediately

↓

Test suite stays honest

-------------------------

Mock

↓

Silently passes when the interface changes

The bug hides until runtime
```

---

## Production Notes

```text
• Prefer Fakes over Mocks for dependencies with behavior.

  Fakes break when interfaces change,
  keeping your test suite honest.
  Mocks silently pass.

• Write Fakes in a testFixtures source set.

  They can be shared between unit tests and integration tests.

• A Fake must implement the same interface as the real class.

  This is WHY constructor injection matters (DOC 14).
  You can only swap a Fake in if the dependency is injected.

• Verify behavior through output state, not method calls.

  Assert on viewModel.uiState.value,
  not on "was refreshUsers() called exactly once?"
```

## A Common Misconception

```text
"Mocks are better because they verify behavior."
```

Mocks verify that a method was called:

```kotlin
verify(mockRepository).refreshUsers()
```

But this tests the implementation, not the behavior.

The test becomes fragile:

```text
Change the internal flow of the ViewModel

↓

The verify() call fails

↓

Even if the screen still works correctly
```

Instead, verify behavior through output:

```kotlin
assertIs<HomeUiState.Success>(viewModel.uiState.value)
```

```text
If the screen shows the right state,

the behavior is correct.

How the ViewModel got there is an implementation detail.
```
