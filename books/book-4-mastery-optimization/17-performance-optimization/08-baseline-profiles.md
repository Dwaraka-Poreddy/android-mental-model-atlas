## The Problem

App startup on first install is slow.

Why?

```text
User taps app icon
        ↓
Zygote forks a new process
        ↓
ART (Android Runtime) loads your classes
        ↓
Classes are interpreted — not yet compiled to native code
        ↓
App starts slowly — first run is always the slowest
```

ART uses a Just-In-Time (JIT) compiler.

Classes start as interpreted bytecode.

Over the first few runs, JIT compiles the hot paths to native code.

This is called the "profile-guided optimization" window.

```text
Run 1: interpreted    → slow
Run 2: partially JIT → medium
Run 3: fully JIT     → fast
```

The problem:

```text
User first opens your app
        ↓
ART interprets code
        ↓
"This app is slow" → 1-star review → uninstall
```

The user who uninstalls never reaches Run 3.

---

## The Mental Model

A Baseline Profile is a pre-compiled list of classes and methods.

ART compiles them to native code during app install,
before the user opens the app for the first time.

```text
Baseline Profile
    ↓
"ART: here are the hot paths.
 Compile these before the first launch."
```

The user's first run is fast
because native code is already ready.

---

## The Difference

```text
Without Baseline Profile          With Baseline Profile

App installed                     App installed
    ↓                                  ↓
User opens app                    ART compiles hot classes
    ↓                             (background, during install)
ART interprets → slow                  ↓
    ↓                             User opens app
Compiles over next few runs            ↓
    ↓                             ART runs compiled code → fast
Run 3: finally fast               Run 1: already fast
```

The user never experiences the slow first run.

## How To Create a Baseline Profile

Baseline Profiles are generated by running a test
that exercises the critical user journeys.

```kotlin
// app/src/androidTest/java/com/example/BaselineProfileGenerator.kt

@ExperimentalBaselineProfilesApi
class BaselineProfileGenerator {

    @get:Rule
    val baselineProfileRule = BaselineProfileRule()

    @Test
    fun startup() = baselineProfileRule.collect(
        packageName = "com.example.gofood"
    ) {
        pressHome()
        startActivityAndWait()
    }

}
```

This generates a `baseline-prof.txt` file.

That file gets packaged into the APK.

The Play Store uses it to pre-compile code during install.

## Production Code

A real Macrobenchmark test measuring startup with and without the profile.

Use this to confirm that the Baseline Profile actually helps.

```kotlin
@RunWith(AndroidJUnit4::class)
class StartupBenchmark {

    @get:Rule
    val benchmarkRule = MacrobenchmarkRule()

    @Test
    fun startupWithoutBaselineProfile() = benchmarkRule.measureRepeated(
        packageName = "com.example.gofood",
        metrics = listOf(StartupTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.COLD,
        compilationMode = CompilationMode.None()    // ← no pre-compilation
    ) {
        pressHome()
        startActivityAndWait()
    }

    @Test
    fun startupWithBaselineProfile() = benchmarkRule.measureRepeated(
        packageName = "com.example.gofood",
        metrics = listOf(StartupTimingMetric()),
        iterations = 5,
        startupMode = StartupMode.COLD,
        compilationMode = CompilationMode.Partial(   // ← uses Baseline Profile
            baselineProfileMode = BaselineProfileMode.Require
        )
    ) {
        pressHome()
        startActivityAndWait()
    }

}
```

### How To Read It

```text
CompilationMode.None()
    ↓
ART interprets everything — no pre-compilation.
This simulates first launch on a fresh install
without a Baseline Profile.
Measures your "before" baseline.


CompilationMode.Partial(BaselineProfileMode.Require)
    ↓
ART pre-compiles the methods in the Baseline Profile.
Simulates first launch on a device where the Play Store
applied the profile during install.
Measures your "after" improvement.


iterations = 5, StartupMode.COLD
    ↓
Kill the process. Launch. Measure.
5 times, then average.
Cold start is the worst case — measures the full startup path.


startActivityAndWait()
    ↓
Waits until the Activity is resumed.
Combine with reportFullyDrawn() in the Activity
to also capture timeToFullDisplayMs —
time until the actual content is shown to the user.
```

## The Profile Generator vs The Benchmark

These are two different things.

Easy to confuse.

```text
BaselineProfileRule (generator)
    ↓
Records which classes and methods are executed
during the test journey.
Produces: baseline-prof.txt
Used: at install time by ART.


MacrobenchmarkRule (benchmark)
    ↓
Measures how long startup actually takes.
Compares: with profile vs without profile.
Used: by developers to verify the profile works.
```

You need both.

The generator creates the profile.

The benchmark confirms the profile helps.

## Adding Profile Installer

For sideloaded APKs (not installed via Play Store),
the Play Store's pre-compilation step doesn't run.

Profile Installer applies the profile at first app launch instead:

```kotlin
// build.gradle.kts
dependencies {
    implementation("androidx.profileinstaller:profileinstaller:1.3.1")
}
```

That is the entire setup.

Profile Installer installs itself and applies the profile
during the first launch after install.

No code changes needed beyond the dependency.

## Production Notes

```text
* Baseline Profiles reduce startup by 20-40% on first launch.
  These are Google's measurements from apps using Compose.
  Your improvement depends on how much code you pre-compile.

* Profile Installer applies profiles even on non-Play-Store installs.
  Internal distribution, sideloaded APKs, Firebase App Distribution —
  all benefit automatically.

* Regenerate Baseline Profiles when you add major new flows.
  The profile reflects the code at generation time.
  If you add a new HomeScreen and don't update the profile,
  the new code runs interpreted on first launch.

* Jetpack libraries ship their own Baseline Profiles.
  Compose, Room, Navigation — all include profiles.
  You get those optimizations for free on first launch
  even before creating your own profile.

* Add Baseline Profile generation to your release process.
  Run it when cutting a release, not during development.
  Profiles are tied to specific APK builds.
```

## Common Misconception

A common misconception is:

```text
"Baseline Profiles only help on first launch."
```

Incorrect.

```text
Baseline Profiles also improve:

    Cold starts after device restart
        ↓
    The system may clear compiled code during low memory.
    The profile allows ART to recompile quickly.

    Cold starts after the OS kills the app
        ↓
    Same situation — process is gone,
    ART uses the profile to restore fast startup.

    Warm starts
        ↓
    Even when the process is alive,
    the profile ensures hot methods are compiled to native.
```

First launch is the most dramatic improvement.

But any cold or warm start benefits from the profile.
